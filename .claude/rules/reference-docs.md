# リファレンスドキュメントの同期

## 更新ルール

ソースコードを修正した際、関連するリファレンスドキュメントの記載内容と差分が生じる場合は、
同じブランチ内でドキュメントも更新すること。

### 対象マッピング（docs/ 配下）

| 変更対象 | 確認すべきドキュメント |
|---|---|
| API の変更（メソッド追加・シグネチャ変更等） | `docs/api-reference.md` |
| アーキテクチャ・処理フローの変更 | `docs/architecture.md` |
| 設定項目の追加・変更 | `docs/configuration.md` |
| エラーコードの追加・変更 | `docs/error-codes.md` |
| トラブルシューティング手順の変更 | `docs/troubleshooting.md` |

### 確認手順

1. 変更内容から関連するドキュメントを上記マッピングで特定
2. API シグネチャ、パラメータ、処理フローに変更があればドキュメントも更新
3. 新規機能追加時は該当するドキュメントへの追記を検討

## ドキュメント作成時のルール

### レビュー必須（コミット前に実施）

ドキュメントを新規作成・大幅更新した場合は、**コミット前に必ずレビューを行う**こと。
文書を書き終えた直後にそのままコミット・プッシュ・PR 作成へ進んではならない。

**ワークフロー**: 文書作成 → **レビュー（下記2点）** → コミット → プッシュ → PR 作成

レビュー観点:
1. ソースコードとの照合（定数値、メソッドシグネチャ、処理ロジック）
2. 他のドキュメントとの整合性確認

### コンテキスト量の管理

ドキュメントの作成は大量のソース読み込みを伴うため、
**コンテキストウィンドウの圧迫に注意**すること。

- 1セッションで扱う範囲を事前に見積もり、オーバーする可能性がある場合は**タスクを細分化**する
- コンテキスト圧縮が発生すると内容の欠落・希薄化が起きるリスクがある
- 大規模な文書は章単位・セクション単位で分割して作成し、後から統合する
