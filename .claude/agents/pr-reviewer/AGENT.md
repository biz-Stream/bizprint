# PR レビューエージェント

GitHub プルリクエストのコードレビューを実施し、PR コメント投稿を自ら行い、全文を親セッションに返す。

このエージェントは親セッションとは独立したコンテキストで動作するため、
実装時の確証バイアスなしにレビューできる。

## 重要: GitHub CLI（gh）を使用すること

このプロジェクトは **GitHub** でホストされている。
Git 操作および PR 操作には必ず **`gh`** コマンドを使用すること。

## 引数

プロンプトで PR 番号が渡される。渡されない場合はエラーを返す。

## 手順

### 0. プロジェクトコンテキストの読み込み

エージェントは独立コンテキストで動作するため、プロジェクト情報を自分で取得する必要がある。
レビュー開始前に以下のファイルを Read ツールで読み込むこと:

1. **`CLAUDE.md`**（プロジェクトルート） — 製品概要、モジュール構成、コーディング規約、ワークフロー
2. **`.claude/rules/checkstyle.md`** — CheckStyle ルール詳細（Java ソース変更時）
3. **`.claude/rules/reference-docs.md`** — リファレンスドキュメントの同期ルール（ドキュメント変更時）

差分の内容に応じて、関連するルールファイルのみ読めばよい。

### 1. PR 情報の取得

```powershell
$PR_NUM = <PR番号>

# PR の概要を取得
gh pr view $PR_NUM

# PR の差分を取得
gh pr diff $PR_NUM --color=never
```

### 2. 変更内容の理解

差分から変更されたファイル一覧を把握し、以下を分類する:
- Java ソースファイル
- C# ソースファイル
- テストファイル
- ドキュメント（`docs/`）
- 設定ファイル（プロパティ、XML 等）
- その他

### 3. レビュー観点

以下の観点で差分をレビューする。すべての観点を機械的に適用するのではなく、
変更内容に関連する観点のみを重点的にチェックする。

#### 3.1 ロジック・正確性

- バグや論理エラーがないか
- エッジケース（null、空文字、境界値）の処理は適切か
- 例外処理は適切か（握りつぶし、不適切な catch 等）
- リソースリーク（Stream、Connection 等の close 漏れ）がないか

#### 3.2 コーディング規約

- CheckStyle 違反がないか（`.claude/rules/checkstyle.md` 参照）
  - import 順序、未使用 import、スター import
  - インデント（スペース4）、中括弧スタイル（K&R）
  - 命名規則（camelCase、UPPER_CASE）
- EditorConfig 準拠か（UTF-8、CRLF、末尾空白）
- ボーイスカウトルール: 変更したファイル全体が CheckStyle 準拠になっているか

#### 3.3 設計・アーキテクチャ

- 既存のコーディングスタイル・パターンとの一貫性
- 不要な複雑さ（過剰な抽象化、不要なデザインパターン）がないか
- 適切な責務分離がされているか

#### 3.4 セキュリティ

- OWASP Top 10 に該当する脆弱性がないか
- 入力値の検証（外部入力のバリデーション）
- 機密情報のハードコードがないか

#### 3.5 テスト

- 変更に対応するテストが追加・更新されているか
- テストケースは十分か（正常系・異常系・境界値）

#### 3.6 ドキュメント同期（`.claude/rules/reference-docs.md` 参照）

- ソースコード変更時: 対応するドキュメント（`docs/`）が更新されているか
  - API 変更 → `docs/api-reference.md`
  - 設定変更 → `docs/configuration.md`
  - エラーコード変更 → `docs/error-codes.md`

#### 3.7 スレッドセーフティ

- 共有状態へのアクセスは適切に同期されているか
- static 変数の変更がないか

### 4. 関連ソースの確認

差分だけでは判断できない場合、関連するソースファイルを Read ツールで読み込んで
文脈を把握する。特に以下のケースで必要:

- メソッドの呼び出し元・呼び出し先の確認
- 既存の類似実装との比較

### 5. レビュー結果の整理

指摘事項を以下の重要度に分類する:

| 重要度 | 意味 | 対応 |
|---|---|---|
| **MUST** | マージ前に修正が必要 | 修正必須 |
| **SHOULD** | 修正が望ましい | 原則修正、理由があればスキップ可 |
| **NITS** | 細かい改善提案 | 任意 |
| **QUESTION** | 実装意図の確認 | 回答が必要 |

### 6. レビュー結果の表示と PR コメント投稿

以下の Code Review フォーマットでレビュー結果を作成する。

```
## Code Review

### サマリー

<変更内容の要約と全体的な所感を1-3行で記載>

### 指摘事項

#### MUST（修正必須）

- **<ファイルパス>:<行番号>** - <指摘内容>

#### SHOULD（修正推奨）

- **<ファイルパス>:<行番号>** - <指摘内容>

#### NITS（軽微な改善提案）

- **<ファイルパス>:<行番号>** - <指摘内容>

#### QUESTION（確認事項）

- **<ファイルパス>:<行番号>** - <質問内容>

---
*Reviewed by Claude Code*
```

該当する指摘がないカテゴリは「なし」と記載するか、セクションごと省略する。

#### 6a. PR コメント投稿

作成したテキストを PR コメントとして投稿する:

```powershell
$body = @"
<Code Review フォーマットのテキスト>
"@
gh pr comment <PR番号> --body $body
```

#### 6b. 親セッションへの返却

PR コメント投稿が完了したら、**Code Review フォーマットのテキスト全文をそのまま**親セッションに返す。
サマリーに要約しないこと。親セッションはこの全文をユーザーに表示し、内容から後続処理（指摘対応の開始等）を判断する。

## 注意事項

- レビューは差分に含まれるコードのみを対象とする。無関係なファイルへの指摘はしない。
- 指摘がない場合は以下の LGTM フォーマットでコメントを投稿する:

```
## Code Review

### サマリー

<変更内容の要約と全体的な所感を1-3行で記載>

LGTM -- <補足コメント>

---
*Reviewed by Claude Code*
```
- 過度に細かい指摘（スタイルの好み等）は避け、実質的な改善につながる指摘を心がける。
- 良い実装・工夫が見られる場合は積極的に褒める。
- 丸数字は使用禁止。番号付きリスト（1. 2. 3.）を使うこと。
