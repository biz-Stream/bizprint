# review-pr スキル

GitHub プルリクエストのコードレビューを実施し、指摘事項を PR コメントとして投稿する。

## 重要: 別セッションで実行すること

このスキルは PR を作成したセッションとは**別のセッション**で実行する。
同じセッションで実行すると、実装時のコンテキストが残っており、
確証バイアスにより問題を見逃すリスクがある。

## 引数

```
/review-pr <PR番号>
```

PR 番号が省略された場合はユーザーに確認する。

## 手順

### 1. PR 情報の取得

```bash
PR_NUM=<PR番号>

# PR の概要を取得
gh pr view $PR_NUM

# PR の差分を取得
gh pr diff $PR_NUM --color=never
```

### 2. 変更内容の理解

差分から変更されたファイル一覧を把握し、以下を分類する:
- Java ソースファイル
- C# ソースファイル
- テストファイル
- ドキュメント（`docs/`）
- 設定ファイル（プロパティ、XML 等）
- その他

### 3. レビュー観点

以下の観点で差分をレビューする。すべての観点を機械的に適用するのではなく、
変更内容に関連する観点のみを重点的にチェックする。

#### 3.1 ロジック・正確性

- バグや論理エラーがないか
- エッジケース（null、空文字、境界値）の処理は適切か
- 例外処理は適切か（握りつぶし、不適切な catch 等）
- リソースリーク（Stream、Connection 等の close 漏れ）がないか

#### 3.2 コーディング規約

- CheckStyle 違反がないか（`.claude/rules/checkstyle.md` 参照）
  - import 順序、未使用 import、スター import
  - インデント（スペース4）、中括弧スタイル（K&R）
  - 命名規則（camelCase、UPPER_CASE）
- EditorConfig 準拠か（UTF-8、CRLF、末尾空白）
- ボーイスカウトルール: 変更したファイル全体が CheckStyle 準拠になっているか

#### 3.3 設計・アーキテクチャ

- 既存のコーディングスタイル・パターンとの一貫性
- 不要な複雑さ（過剰な抽象化、不要なデザインパターン）がないか
- 適切な責務分離がされているか

#### 3.4 セキュリティ

- OWASP Top 10 に該当する脆弱性がないか
- 入力値の検証（外部入力のバリデーション）
- 機密情報のハードコードがないか

#### 3.5 テスト

- 変更に対応するテストが追加・更新されているか
- テストケースは十分か（正常系・異常系・境界値）

#### 3.6 ドキュメント同期（`.claude/rules/reference-docs.md` 参照）

- ソースコード変更時: 対応するドキュメント（`docs/`）が更新されているか
  - API 変更 → `docs/api-reference.md`
  - 設定変更 → `docs/configuration.md`
  - エラーコード変更 → `docs/error-codes.md`

#### 3.7 スレッドセーフティ

- 共有状態へのアクセスは適切に同期されているか
- static 変数の変更がないか

### 4. 関連ソースの確認

差分だけでは判断できない場合、関連するソースファイルを Read ツールで読み込んで
文脈を把握する。特に以下のケースで必要:

- メソッドの呼び出し元・呼び出し先の確認
- 既存の類似実装との比較

### 5. レビュー結果の整理

指摘事項を以下の重要度に分類する:

| 重要度 | 意味 | 対応 |
|---|---|---|
| **MUST** | マージ前に修正が必要 | 修正必須 |
| **SHOULD** | 修正が望ましい | 原則修正、理由があればスキップ可 |
| **NITS** | 細かい改善提案 | 任意 |
| **QUESTION** | 実装意図の確認 | 回答が必要 |

### 6. PR コメントの投稿

レビュー結果を PR コメントとして投稿する。

```bash
gh pr comment $PR_NUM --body "$(cat <<'EOF'
## Code Review

### サマリー

<変更内容の要約と全体的な所感を1-3行で記載>

### 指摘事項

#### MUST（修正必須）

- **<ファイルパス>:<行番号>** - <指摘内容>

#### SHOULD（修正推奨）

- **<ファイルパス>:<行番号>** - <指摘内容>

#### NITS（軽微な改善提案）

- **<ファイルパス>:<行番号>** - <指摘内容>

#### QUESTION（確認事項）

- **<ファイルパス>:<行番号>** - <質問内容>

---
*Reviewed by Claude Code*
EOF
)"
```

該当する指摘がないカテゴリは「なし」と記載するか、セクションごと省略する。

### 7. 投稿前の確認

**PR コメントを投稿する前に、必ずユーザーにレビュー内容を提示して確認を取る。**
ユーザーが内容を承認してから投稿する。

## 注意事項

- レビューは差分に含まれるコードのみを対象とする。無関係なファイルへの指摘はしない。
- 指摘がない場合は「問題なし」のコメントを投稿する（LGTM）。
- 過度に細かい指摘（スタイルの好み等）は避け、実質的な改善につながる指摘を心がける。
- 良い実装・工夫が見られる場合は積極的に褒める。
